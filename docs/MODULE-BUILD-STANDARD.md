# 售前知识模块建设标准

本标准由 RAG、Agent、Prompt Engineering 三个深度模块及全库整合实践共同提炼，适用于所有云计算 × AI 平台售前知识模块。它是编辑与验收规范，不得作为章节、卡片、页脚或提示语展示给读者。

历史反馈的追溯关系、永久防复发机制和新模块 Definition of Done 统一维护在 `MODULE-QUALITY-GATES.md`。建设时必须同时执行两份内部规范；不得只修当前模块而不更新通用门禁。英文版同时执行 `ENGLISH-EDITORIAL-STANDARD.md`，以一手来源重建专业表达，不以中文句子作为回译底稿。

## 1. 建设目标

每个模块都应帮助有 Python / API 基础的技术售前人员完成一条完整判断链：

> 讲清概念 → 理解机制 → 判断边界 → 连接云服务 → 形成方案 → 回答客户 → 用证据校验

模块不是术语百科、论文复述或产品宣传页。读者完成模块后，至少应能够：

- 用准确但非学术化的语言向客户解释主题是什么、为什么有价值；
- 判断客户问题是否适合使用该能力，以及何时不适合；
- 识别方案所需的模型、数据、算力、安全、集成与运营能力；
- 提出下一轮客户发现问题，并定义可验证的 PoC 或上线门槛；
- 区分事实、工程判断、厂商能力和仍需客户实测的假设。

HTML 不是 PPT 的逐页搬运，也不是把幻灯片纵向拼成长页。PPT 可以提供内容地基和查漏线索；HTML 必须进一步利用可检索入口、稳定深链接、章节导航、渐进阅读、机制 / 因果视图、情境问答和响应式呈现，让读者更快找到、理解、判断和复用知识。若网页只改变载体而没有增加这些阅读能力，不得视为“在 PPT 地基上更进一步”。

## 2. 读者、语言与理论深度

### 2.1 读者基线

- 默认读者有 Python / API 基础，但不要求掌握研究数学、模型训练或底层分布式系统实现。
- 内容重概念、机制、边界和方案判断；理论服务于售前决策，不以推导完整性为目标。
- 关键数字必须同时给出适用数据集、实验条件、时间或产品版本和不可外推的边界。

### 2.2 语言规则

- 正文以中文为主，专业术语首次出现写为“中文（English，缩写）”或“中文 · English”；后文按语境使用中文或通行缩写。
- 英文必须使用领域通行名称，不采用逐词机器翻译。英文版共享模块、概念与来源身份，但正文从原始来源和技术机制独立撰写，并按 `ENGLISH-EDITORIAL-STANDARD.md` 独立审校。
- 中英文模块共享同一 `knowledgeView`、宏观章节顺序和交互能力。英文可以独立组织句子和解释，但不能跳过中文版首屏机制视图、另造额外通用章节，或让同一模块形成不同的学习路径。
- 面向读者的中文优先使用具体名词和日常动词。`客户信号`、`来源台账`、`知识供应链`、`责任链`、`运营闭环`、`责任闭环` 等内部或咨询式简称不得作为公开标题、标签或解释；分别按语境改写为客户遇到的情况、来源与证据资料库、资料处理流程、责任分工、后续维护或持续改进流程。
- `Control Plane`、`Terminal State`、`Baseline`、`Mapping`、`Governance`、`Observability` 等词只有在技术区别确实重要时保留。首次出现必须先给普通中文解释，例如“共同使用的控制部分（Control Plane）”“最终状态（Terminal State）”“与现有做法对比（Baseline）”和“用日志、指标与链路追踪观察系统（Observability）”。产品选择场景用“对应到”或“选择”，向量、身份与权限关系才使用“映射”。
- 避免让相邻段落或不同模块反复套用“不是 A，而是 B”“先……再……最后……”等固定句式。定义、机制、方案和边界应按各自问题选择最自然的表达。
- 标题采用客观陈述或名词短语，不使用“先确定”“我们来看”“你需要”“不要一上来”等编辑者对读者说话的表达。
- 数字与量词、缩写与释义、专有名词作为完整语义单元排版，不允许产生只有少量汉字的孤行。

### 2.3 深度标尺

每个核心知识点按以下顺序展开，深度到“能够支持客户判断”为止：

1. **定义**：它是什么，解决什么问题；
2. **机制**：输入、关键处理、输出以及为什么有效；
3. **边界**：它不解决什么，常见误解和失效条件；
4. **判断**：售前据此应该问什么、选什么或验证什么；
5. **证据**：结论由什么来源支持，来源不能证明什么。

禁止用概念堆叠代替解释，也禁止为了显得深入而加入不会改变售前判断的论文细节。

### 2.4 外部参考资料的使用边界

- `external_reference/` 是原始材料投放区，不是公开事实台账；材料可用于补全概念、关系、客户问题和讲解顺序。
- 外部材料不是模块模板。不能沿用 PPT 目录作为网页目录，也不能用“已覆盖全部幻灯片主题”替代知识完整性判断。
- PPT 的章节顺序、每页构图、卡片数量和信息密度均不自动继承到 HTML。网页目录应按搜索入口、客户任务、机制理解和决策路径重组。
- 模块结构应从客户需要完成的判断链反推：原理如何工作、为什么失败、怎样选型、如何连接云服务、如何验收和怎样回答客户。PPT 只参与查漏。
- PPT、白皮书或二次整理材料中的动态数字、产品能力、协议版本、价格、法规和性能比较不得直接升级为公开结论。
- 公开结论应回到论文、标准、协议规范或厂商官方文档核验，并只在 `app/reference-content.mjs` 登记链接、证据类别、边界和核验日期。
- 若外部材料提供了有价值但暂时无法核验的观点，可保留为编辑线索，不进入读者页面，也不伪装成确定事实。
- 发布前应做一次“独立增量”复核：页面是否解释了外部材料未展开的因果机制、生产失败、方案权衡或客户决策，并提供网页原生的检索、导航、渐进展开和可回到问题的交互；若只是换一种排版复述材料，不得发布为完成模块。

### 2.5 机制与组件完整性

如果一个主题存在行业通行的生命周期、动作循环或核心组件，正文不能只在总览图、标签或一句话中提到它们。每个关键动作和组件都必须分别讲清：

- 它在整体机制中的职责，以及与前后步骤的输入输出关系；
- 它如何工作，依赖哪些模型、数据、状态、工具或控制面；
- 常见失败是什么，怎样从日志、轨迹或业务结果识别；
- 哪些规则必须由确定性应用、身份系统或人工审批执行；
- 能连接哪些云能力，售前应追问什么，并用什么结果验收。

同一词在不同框架中的定义或粒度不一致时，应明确说明本模块采用的教学口径及其边界，不能把一种实现方式写成唯一标准。章节是否“讲过”以读者能否据此解释机制、判断方案和回答客户为准，不以关键词是否出现为准。

## 3. 模块信息架构

以下是稳定的知识骨架，不是固定章节数，也不是页面数量配额。允许按主题合并、拆分或省略，但不能遗漏该主题真正需要的判断链。

| 知识区块 | 必须回答的问题 | 可选呈现方式 |
| --- | --- | --- |
| 模块定位 | 该能力是什么；在方案中负责什么；不负责什么 | 一句话定位、核心边界 |
| 相关模块 | 哪些概念由其他模块主要讲解；当前模块为何需要它 | 可导航关系卡、局部解释 |
| 基础概念与原理 | 核心术语、输入输出、工作机制是什么；通行动作循环与关键组件如何协作 | 因果链、流程、对比卡、示例 |
| 架构、模式与变体 | 常见实现如何组合；复杂度为何不同 | 架构链、模式对比、责任边界 |
| 适用边界与选型 | 什么时候采用；什么时候不用；与替代方案如何比较 | 决策树、适用性表、反例 |
| 云服务连接 | 能力落在哪些云服务；怎样形成可销售的整体方案 | 阶段映射、发现问题、方案组合 |
| 工程保障 | 如何评估、安全上线、观测、治理成本与持续运营 | PoC 清单、指标分层、风险卡 |
| 客户实战包 | 客户会怎么问；怎样回答并继续发现需求 | 高频问题与深度回答 |
| Reference 回链 | 结论依据在哪里；核验日期和外推边界是什么 | 稳定 `sourceId` 链接到 `/references` |

### 3.1 网页原生阅读链

正式模块必须同时支持“快速定位”和“连续精读”，不能把所有内容同权重一次性铺开：

- 首页或全局入口可检索模块、专业术语、章节、高频问题和实战练习；搜索结果必须指向稳定模块或章节地址，不能只是前端装饰。`/questions` 从正式内容注册表派生全部客户问题，支持关键词、模块与类别组合查询，并回链原模块的具体问题锚点，不得维护第二份问题清单。
- 模块提供可见章节目录；桌面端在阅读时保持可达并标示当前位置，移动端提供等价的章节入口，不让读者依赖浏览器长距离滚动条猜位置。
- 每章先给结论、机制或判断，再按需展开案例、失败、证据和延伸解释。重要边界、关键前提和风险门槛不得默认折叠。
- 核心流程、状态、层级、责任传递或因果关系应选择适合的机制视图表达；不能把本可形成心智模型的内容全部压成同样的卡片或段落。
- 重要章节应直接放置与该知识点有关的客户情境问题，或提供能定位到对应问答并返回原章节的入口；不得让读者读完原理后再到长问答列表中重新寻找。
- 共享模块必须给出可观察的学习结果、由心智模型到方案判断再到证据验收的推荐路线，以及能产生真实方案产物的实战任务。实战任务包含情境、行动、产物、通过标准和稳定 `sourceId`，不能只写“阅读本章”或复述概念。
- 推荐路线的步骤数由主题复杂度决定；共享标题使用“建议学习顺序”等不绑定数量的表述，不能在路线扩展后仍显示“固定三步”。
- 共享模块还要有完整课程地图：每个主题同时说明机制解释、会改变的售前判断、关键边界和题内来源。课程地图补齐知识版图，不能与后续原理卡、深挖或问答重复堆字。
- 学习结果、推荐路线、课程地图和实战任务是内容契约，不是必须各占一个公开大区块的版式契约。`readingProfile: "focused"` 的模块应把这些内容编入一条模块独有的连续论证：例如诊断链、责任链或决策门；只要稳定锚点、完整内容、全局检索与验收产物仍可到达，就不得为了统一目录重新拆成“学习—课程—原理—选择”的重复章节。
- `knowledgeView` 决定模块最重要的关系怎样表达，`readingProfile` 决定公开阅读节奏；共享壳只负责导航、术语、证据、问答和响应式能力，不能强迫不同主题具有相同章节数量、首屏计数或卡片密度。
- 外部讲义只用于识别覆盖面与学习难点；学习路线按网页阅读和客户决策重新组织，不沿用讲义章节顺序。公开页面不展示材料对比、编辑选择或“从 PPT 改写”等内部过程。
- 交互只用于搜索、比较、展开细节、切换真实视角、追踪证据或推进方案判断。没有信息增益的轮播、动画、悬停效果和伪按钮不得进入正式页面。
- 所有关键内容必须在匿名、无个人状态的情况下可访问；交互增强不能破坏稳定 URL、打印阅读或无脚本时的基本内容链。

### 3.2 按需原则

- 不为图、卡片、案例、问答或来源设置数量指标；内容数量由知识完整性决定。
- 图示只在它比文字更容易说明流程、层级、映射或状态变化时使用，不“为了放一张图”而放图。
- 同类卡片只有一个时，可以改用正文或强调条；内容不足以形成真实比较时，不制作比较表。
- 不展示编辑原则、语言策略、构建信息、维护流程、内部字段或发布门槛。页面只保留对读者学习和客户交流有直接价值的内容。

### 3.3 新增内容的更新时间

- 自 2026-07-20 起，模块的事实、机制、边界、判断或回答发生实质变化时，刷新 `app/module-publication.mjs` 中该模块的 `updatedAt: "YYYY-MM-DD"`。模块页只在顶部或底部显示一次“最近更新于 YYYY-MM-DD”。
- 课程章节、独立深挖块、实战任务和证据卡不设置、也不显示逐条更新时间；它们的变化统一反映在模块级日期。
- 只有整道新增的客户问答设置 `addedAt: "YYYY-MM-DD"`，并在模块问答与全站问题查询页一致显示“新增于 YYYY-MM-DD”。后续修改既有问题的答案、证据或措辞时，只刷新模块日期，不改变 `addedAt`。
- 纯排版、样式和格式调整不刷新模块日期。2026-07-20 之前已有模块和问题不回填日期，避免把功能上线时间误写成内容更新时间。
- 模块最近更新时间只说明知识内容何时修订，不替代 `/references` 的来源核验日期、复核周期和 freshness 发布门禁。
- `app/module-publication.mjs` 内部登记模块首次发布日 `introducedAt` 与日期策略生效前无日期问答的稳定身份集合摘要 `legacyUndatedQuestionSetSha256`，两者不进入读者页面。策略生效后引入的新模块，其初始问答必须全部使用模块 `introducedAt` 作为 `addedAt`。
- 身份摘要的规范算法是：只取无 `addedAt` 的问题文本，逐项执行 Unicode NFKC、首尾去空白并把连续空白折叠为单个空格，按代码单元排序，对所得字符串数组的 JSON 表示计算 SHA-256。它绑定历史问题身份而非数量，删除旧题后补入同数量无日期新题也会失败。
- 只有明确改写或删除历史问题并完成内容审计时，才能同步刷新 `legacyUndatedQuestionSetSha256`；不能用刷新摘要来容纳漏填 `addedAt` 的新增问题。

## 4. 相关模块串联

页面统一使用“相关模块”，不用“模块依赖”。每个跨模块概念只设一个主要归属，当前模块提供完成当前理解所需的局部解释，并链接到主要模块，避免复制全文和版本漂移。

相关模块条目建议使用以下字段：

```js
{
  concept: "当前需要理解的概念",
  owner: "主要归属模块",
  href: "/modules/stable-slug",
  relation: "前置知识 / 生成约束 / 工程控制 / 下游演进等",
  local: "它与当前模块的具体关系和局部解释"
}
```

验收要求：

- 链接必须可导航，不能使用无对应页面的装饰性标签；
- `local` 必须解释关系，不能只写“了解更多”；
- 同一术语在不同模块的定义、英文名称和边界必须一致；
- 模块间形成双向关系时，双方都应在下一次相关更新中复核，但不要求为了对称而增加无价值卡片。

## 5. 云服务销售连接

凡主题能够连接云服务，都要说明它在整体方案中的落点。连接不是罗列产品名，而是建立“技术环节—云能力—客户价值—发现问题—验收指标”的因果链。

建议字段：

```js
{
  stage: "技术或业务阶段",
  services: "云服务能力类别，不限单一厂商产品",
  value: "该能力为什么对客户方案有价值",
  discover: "售前下一步应该向客户追问什么",
  acceptance: "需要由客户基线或 PoC 验证的指标（按需）",
  responsibility: "云平台、客户应用、数据 / 身份系统与人工各自负责什么"
}
```

当正文进一步点名具体云产品、版本或购买选项时，还必须通过稳定 `sourceId` 关联当期官方资料，并按该事实的实际可得性记录：

```js
{
  capability: "实际使用的产品能力，不只写营销名称",
  lifecycle: "GA / Preview / Deprecated 等官方状态",
  regions: "适用地域，或明确要求采购时点重新核验",
  limits: "配额、模型 / 硬件、网络、数据或功能限制",
  pricingUnit: "按 token、请求、时长、实例、存储或其他计价单位",
  sourceId: "stable-source-id"
}
```

页面展示的核验日期必须通过 `sourceId` 从 `app/reference-content.mjs` 解析，不在模块内容中复制 `verifiedAt`、来源标题、URL、证据类别或来源边界。

内容要求：

- 覆盖与主题真实相关的数据、模型、计算、集成、安全、网络、可观测与 FinOps 能力，不机械填满所有类别。
- 区分技术可行性与产品可用性；地域、价格、配额、SLA、型号和产品规格必须以当期官方资料为准。
- 区分“云平台提供的能力”和“客户仍需实现的业务责任”。托管运行、身份、网关、评估或向量检索不自动完成数据治理、业务授权、结果正确性、补偿和人工责任。
- 不确定或快速变化的状态、地域、限制和价格不得用模糊的“支持”概括；应写明核验日期、需重新核验的采购条件和不能承诺的边界。
- 不把某个协议、模型或单一产品描述成完整平台能力；明确它在方案中的责任边界。
- 把“能卖什么”转成客户价值和验收方式，避免只形成云产品清单。

## 6. 客户问答、证据卡与 Reference 数据契约

### 6.1 客户高频问题

每道问题至少包含：

```js
{
  q: "客户可能直接提出的问题",
  a: "可以先说出口的结论短答",
  depth: "机制、权衡、边界与实施含义",
  ask: "用于继续发现需求的售前下一问",
  tag: "问题类别",
  basis: "证据类型或判断基础",
  evidence: [
    {
      sourceId: "stable-source-id",
      supports: "该来源具体支持什么，以及不能支持什么"
    }
  ],
  addedAt: "YYYY-MM-DD" // 2026-07-20 起整道新增的问题必填；后续改写不刷新
}
```

要求：

- `a` 先回答问题，不先铺背景；`depth` 必须比短答多一层机制、权衡或边界。
- `ask` 必须能推进需求发现、方案选择或 PoC 设计，不能重复问题。
- `supports` 不能只写来源名称，必须说明证据作用和外推限制。
- 客户问题必须覆盖模块正文中的重要边界、云服务机会与常见反对意见；正文变化时同步检查相关问答。
- 问题数量由模块需要覆盖的独立客户决策面自然产生。若多个模块异常地收敛为同一题数，必须回查是否使用了统一追加配额；修复方式是补齐或合并真实主题，而不是换成另一个统一数字。

### 6.2 证据卡

证据卡用于展示真正帮助售前判断的关键数据，不是来源数量展示。建议字段：

```js
{
  metric: "数据或定性结论的突出部分",
  title: "指标名称",
  finding: "来源在什么条件下观察到什么",
  boundary: "不能如何外推，客户还要验证什么",
  sourceId: "stable-source-id"
}
```

没有可靠数字时可以使用定性证据，不得为了填卡片而制造数字、混合不同实验条件或隐藏厂商实验属性。

### 6.3 统一 Reference 台账

所有来源元数据只在 `app/reference-content.mjs` 维护；模块正文、问答和证据卡只保存稳定 `sourceId`，不得复制来源 URL。

来源记录契约：

```js
"stable-source-id": {
  grade: "证据等级",
  kind: "论文 / 标准 / 官方文档 / 厂商实验 / 行业指南等",
  shortTitle: "页面短标题",
  title: "来源完整标题",
  note: "支持范围、实验条件与外推边界",
  verifiedAt: "YYYY-MM-DD",
  href: "https://..."
}
```

每个模块还必须在 `referenceModules` 中建立唯一分组，包含稳定模块 `id`、中英文名称、模块地址和去重后的 `sourceIds`。所有来源都必须归入至少一个模块分组，页面统一链接到 `/references#source-<sourceId>`。

证据等级表示来源类型与可信程度，不代表结论可以跨模型、数据集、客户、地域或产品版本直接外推。

## 7. 公式与理论深度门槛

公式必须同时满足以下条件才进入模块正文：

1. 公式会直接改变架构、选型、指标或风险判断；
2. 不使用公式会导致关键机制被错误理解；
3. 能先用中文直观解释，再定义变量和业务含义；
4. 能明确说明它不代表什么，以及客户项目仍需实测什么。

若任一条件不满足，改用流程、因果关系、例子或对比说明；论文推导放入 Reference 或延伸阅读。禁止把完整概率推导、训练目标或证明过程作为“内容有深度”的替代品。

发布前对每个公式追问：**读者理解公式后会作出什么不同的售前判断？** 没有具体答案就删除公式。

## 8. 网页原生可读性、交互与动态布局

- 卡片数量由内容数据决定，不把“8 张”“4×2”或任何固定总数写入组件前提。
- 桌面端使用均衡分行算法，每行不超过设计上限，各行数量差不超过 1；例如数量变化时自动形成均衡行，而不是任由浏览器出现明显残缺的尾行。
- 平板优先双列，单数尾项占满最后一行；手机单列并保持原阅读顺序。
- 空列表不渲染空容器；一项、单数、双数和未来增减都必须通过自动检查。
- 同类卡片保持统一的色条、边框、标题位置和强调层级。重要边界使用高对比警示组件，不能与普通说明使用相同视觉权重。
- 连续中文正文、问答正文、表格正文和机制解释在桌面、平板与手机均不得低于 16px；建议桌面使用 17–18px。次级说明不得低于 14px，只有不承载理解责任的编号、眉题或元数据可降至 12px，禁止用 9–11px 承载知识内容。
- 正文行高建议保持 1.65–1.85，连续阅读宽度约 42–48 个汉字；标题与正文的比例服务阅读层级，不以超大标题挤压首屏信息。
- 笔记本是当前主要学习界面：桌面复核优先覆盖约 1366×768 与 1440×900，确保 Hero 不把第一段真实知识完全推到首屏之外，阅读栏不过度挤压正文，长表格与实战任务无需缩小字号才能阅读；同时保持平板和手机完整可用。
- 大字符集中文字体不得通过 `next/font` 或响应头在首屏预加载全部 unicode 分片；优先使用系统中文字体栈，若确需 Web Font，只加载当前页面实际需要的字形，并以全新浏览器直达时正文可立即出现为验收条件。
- 中英文混排、长术语和常见桌面 / 平板 / 手机宽度都要避免异常断行；触控目标、焦点样式、键盘顺序和当前章节状态必须可辨认。
- 专业术语在统一术语表维护中文名、英文名、通行缩写、一句话说明和相关模块；首页只展示跨模块高频入口，完整集合由 `/glossary` 按知识关系组织。高密度区域的术语提示在桌面支持悬停和键盘聚焦，手机与平板支持点击展开；不逐词制造气泡，也不能把关键定义藏在仅 hover 可见的内容里。
- 证据卡以“事实标签—结论—适用边界—来源”为阅读顺序。事实标签不能使用封面级字号压过结论；结论正文不得低于 16px，边界说明不得低于 14px，卡片高度由内容决定而不是用大面积空白强制等高。
- 长模块必须有 sticky 章节导航或等价的持续可达导航；锚点要为粘性页头预留滚动位置，并在桌面与移动端保持相同章节语义。
- 渐进展开必须有可访问名称、键盘和触控行为，并保留可分享的章节或问题锚点。不得显示未真正实现的快捷键、筛选或切换控件。
- 表格只用于两个或更多对象在共同维度上的真实比较。单对象说明、流程和松散事实使用正文、列表或机制视图；移动端优先转为带字段名的语义卡片，确需横向表格时提供明确滚动提示。
- 动画和状态切换应遵守 `prefers-reduced-motion`，不能依赖 hover 才能获得关键内容；交互失效时仍应保留正文、章节和 Reference 基本路径。

依赖 12 轨 CSS Grid 的均衡布局必须复用 `balanceGridRows(items, maxColumns)` 与 `gridSpan(row.length)`，不要直接计算 `12 / row.length`，也不要重新编写只覆盖当前数量的 CSS 特例。普通非 Grid 分组才使用 `balanceRows`。所有 `--*-span` 样式必须提供通栏 fallback，漏传变量时优先保持可读而不是压成窄列。

## 9. 维护与时效性

- `verifiedAt` 表示来源最后核验日，不等同于内容永久有效。具体复核周期与事实生命周期按 `CONTENT-MAINTENANCE.md` 执行。
- 原理、标准、协议、模型目录、云产品能力、地域、价格、配额、安全清单和 SLA 按各自变化速度维护，不使用一个统一页脚日期代替复核。
- 来源变化时先识别受影响结论，再同步复核模块正文、相关模块、客户问答、证据卡和 Reference 分组。
- 产品能力或数字无法确认时，明确写为当期资料 / 客户基线 / PoC 待验证项，不把推断发布为确定事实。
- 新版本保留稳定 slug 与 `sourceId`；来源替换、结论撤回和边界变化应可追踪，不能静默改变原结论含义。
- 模块版本号、核验日期和必要的来源边界可以公开；内部巡检流程、责任人、字段 schema、发布步骤和构建状态不得公开。

## 10. 建设流程

1. **界定范围**：写出一句话定位、明确不覆盖的内容和相关模块。
2. **建立证据底座**：优先核验论文、标准和官方文档，在统一台账创建 `sourceId` 与模块分组。
3. **搭建判断链**：按定义、机制、边界、售前判断、证据组织核心原理。
4. **连接云服务**：把技术环节映射为云能力、客户价值、发现问题与验收指标。
5. **编写实战包**：从真实客户疑问出发，补齐短答、深答、追问、依据和边界。
6. **同步相关内容**：检查相关模块定义、证据卡、问答和 Reference 是否一致。
7. **验证页面**：检查检索入口、sticky 章节导航、渐进阅读、情境问答回链、路由、动态布局、响应式、术语与内部信息隔离。
8. **发布复核**：构建、自动检查、代码检查通过后再推送和公开发布。

### 10.1 发布注册

知识地图收录不等于正文已发布。正式模块完成后，按以下顺序登记：

1. 建立独立 `/modules/<slug>` 正式页面；
2. 在统一术语表登记首次中英对照；
3. 建立 Reference 分组并完成来源核验；
4. 完成客户问答、题内依据和按需证据卡，并在 `app/module-content-registry.mjs` 登记；
5. 在 `app/module-publication.mjs` 登记 path、主标题、术语和内容契约；
6. 运行通用质量检查并完成人工内容评审。

首页入口、专用路由判断和测试名单只从发布注册表派生，不再分别手工维护。全站只保留一个面向读者的知识关系入口：公开动态图谱负责关系探索，首页知识地图负责全量层级浏览；图谱覆盖率、共享术语主干和健康检查保留为后台质量门禁，不另设一个功能重叠的公开总览页。

- [ ] 数字结论包含条件、来源和不可外推边界。
- [ ] 重要边界醒目，普通说明没有抢占视觉层级。

- [ ] 来源只在统一 Reference 台账维护 URL、类别、边界和核验日期。

- [ ] 页面先呈现结论与判断，再按需展开细节；重要边界不被折叠，情境问答可以定位并返回对应章节。

- [ ] 高频术语在首次阅读入口可通过桌面悬停 / 聚焦和触屏点击查看全称与简短说明，完整术语可进入 `/glossary` 查找；正文仍保留可独立理解的定义。
- [ ] 相邻章节没有反复套用“不是 A，而是 B”“先……再……最后……”等固定句式。
- [ ] 证据卡的事实标签没有压过结论与边界；结论、适用条件和来源能按自然顺序连续阅读，没有用超大字号和固定高度制造空白。
- [ ] 首屏没有全量预加载大字符集中文字体；使用全新浏览器标签直达首页和代表模块时，正文无需等待字体分片即可出现。

- [ ] 已优先检查 1366×768 与 1440×900 的笔记本视口，Hero、sticky 阅读栏、正文行宽、表格和实战任务均适合连续学习。
- [ ] 页面不含编辑说明、语言规范、维护规则、内部字段、构建或发布信息。
- [ ] 页面匿名直达，不含 Login、密码输入、个人会话或本机绝对路径；源文件和必要资产均在本仓库。
- [ ] 标题、重要边界和动态卡片使用共享实现，没有模块 ID CSS 特判或固定总数分支。

## 12. 建议的代码级自动检查

后续模块应将以下检查抽成通用测试或数据校验器，而不是为 RAG 单独复制断言：

1. **路由完整性**：知识地图中的每个 slug 唯一、符合 kebab-case，且 `/modules/<slug>` 返回 HTML 200。
2. **首页边界**：首页包含模块入口但不包含任一模块正文、问答、完整来源条目或内部说明。
3. **术语与核心内容**：模块页包含模块中英文名称、核心机制、关键边界、云服务连接和实战包标题。
4. **问答契约**：每项 `q / a / depth / ask / tag / basis` 非空；`evidence` 非空、题内 `sourceId` 去重，页面题内依据数量与数据数量一致。
5. **证据卡契约**：每项 `metric / title / finding / boundary / sourceId` 非空；不允许把卡片数量写成验收值。
6. **来源解析**：所有正文、问答和证据卡 `sourceId` 都唯一解析到 `sourceLedger`，并归入当前模块 Reference 分组。
7. **来源唯一维护**：来源 URL 只出现在 `app/reference-content.mjs`；内容文件和页面组件不得包含外部 URL。
8. **来源字段**：`href` 为 HTTPS，`verifiedAt` 为有效 `YYYY-MM-DD`，必填字段完整；URL、模块 ID、模块 href 和分组内 `sourceId` 不重复。
9. **Reference 覆盖**：台账中每个来源至少属于一个模块分组，`/references` 存在唯一 `source-<sourceId>` 锚点和核验日期。
10. **相关模块**：关系卡 href 能解析到知识地图模块，使用“相关模块”而非“模块依赖”，且不存在指向当前页的无意义自链接。
11. **动态布局**：对 0–12 个或更广数量范围验证 `balanceGridRows` 保序、每行不超过上限、行间数量差不超过 1，且 `gridSpan` 始终返回可被轨道总数整除的整数；响应式规则必须依据正文容器宽度覆盖双列、单数尾项通栏和手机单列，不能只依据浏览器 viewport。
12. **理论深度护栏**：按模块配置禁止出现已确认无售前价值的论文公式、推导术语或公式样式类；同时检查核心直观机制仍存在，避免“删公式也删原理”。
13. **内部信息隔离**：生产 HTML 不包含 `BUILD BRIEF`、编辑原则、语言规范、维护流程、内部字段名、构建信息或本机绝对路径。
14. **可访问与导航**：页面只有一个主标题；关键区块有可访问名称；页内导航、Reference 和相关模块链接均可达；移动端导航不被隐藏。
15. **网页原生入口**：正式模块可从全局检索定位到稳定模块 / 章节 / 问题地址；搜索、快捷键、筛选与章节控件不得只显示而未实现。
16. **章节与渐进阅读**：长模块登记稳定 section ID；章节目录覆盖全部主要区块，当前章节可识别；重要边界不在默认折叠区。
17. **字号下限**：生产样式中承载知识的正文、问答和表格正文不低于 16px，次级说明不低于 14px；更小字号只允许用于非必要元数据。
18. **表格适用性与移动端**：人工确认表格存在共同比较维度；移动端使用语义卡片或提供清晰横向滚动提示，页面不得出现全局水平溢出。
19. **交互真实性**：每个搜索、筛选、标签、快捷键和切换控件都有实际状态变化、键盘 / 触控路径与可访问名称；交互失败不切断正文和来源路径。
20. **云产品事实契约**：点名具体云产品时校验能力、生命周期状态、地域、限制、计价单位、责任边界和 `sourceId`，并通过该 `sourceId` 从统一台账解析 `verifiedAt`；快速变化字段超过复核周期即阻止发布。
21. **人工视觉 QA 记录**：桌面、平板和移动端至少各检查一个代表宽度，并覆盖最长标题、最多字段表格、展开问答、粘性导航与键盘焦点；自动布局测试通过不能豁免人工检查。
22. **术语解释**：进入首页、术语库或术语提示的 termId 必须在统一术语表同时具备中英文名称、简短说明和相关模块，确有通行缩写时再登记缩写；正式提示输出原生可聚焦、可点击的语义控件，且提示内容不替代正文定义。
23. **证据阅读层级**：共享证据卡保持“事实标签—结论—边界—来源”顺序；样式检查事实标签不用封面级字号，结论与边界分别满足 16px / 14px 下限，卡片不设置强制大高度。
