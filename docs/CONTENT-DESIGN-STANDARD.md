# 售前知识库内容与构图规范

本规范适用于当前 19 个正式模块及后续调整。它约束最终阅读页面，不作为页面中的说明内容展示。模块可采用主题定制页面或共享能力组件，但不得为了形式统一而牺牲知识结构。

模块的完整建设与验收契约以 `MODULE-BUILD-STANDARD.md` 为准；本文只补充内容语气与视觉构图约束，避免两处重复维护同一规则。

本规范以“HTML 应在 PPT 内容地基上增加网页能力”为前提。PPT 的页面顺序、卡片构图、信息密度和视觉比例不是网页模板；HTML 必须通过检索、导航、渐进阅读、机制视图、情境问答、证据追踪和响应式呈现提高知识的可发现性与可复用性。

## 1. 知识库语气

- 标题采用客观陈述或名词短语，例如“RAG 适用性判断”“RAG 双链参考架构”。
- 避免把编辑者与读者的对话写进标题：不用“先确定”“再判断”“我们来看”“你需要”“不要一上来”等表达。
- 流程步骤可以使用动词，但必须描述系统动作或客户动作，而不是编辑指令。
- 正文以中文为主；专业术语首次出现提供准确英文或通行缩写。

## 2. 信息层级

- 一级模块标题说明主题；二级标题说明知识对象；卡片标题说明单一概念。
- “重要边界、风险、错误前提”使用高对比警示组件；普通补充说明不得与其争夺视觉层级。
- 编辑规则、语言策略、构建说明等内部信息不进入读者页面。
- 同一概念只设一个主要归属模块；其他模块给出必要的局部解释和回链。
- 原理默认用概念、因果链、边界和客户判断来讲清；仅当公式直接帮助售前做架构、选型或风险判断时展示，否则放入 Reference 或延伸阅读。
- 首页只保留全局入口、知识地图与学习路径；每个模块使用独立页面，不把模块正文重新堆回首页。
- 知识地图只保留层名称与可导航模块；不为每层追加泛化、重复的说明文字或售前提示。
- 所有来源集中在独立的 `/references` 页面，并按模块分组；模块正文不重复展示完整来源台账。
- 页面首先建立读者的心智模型和判断顺序，再选择卡片、图、表或交互；不能先套版式，再把知识拆碎填入。

## 3. 网页原生导航与渐进阅读

- 首页或全局入口提供可检索入口，至少能从模块、专业术语、章节和客户问题进入稳定地址。只过滤当前卡片但标成“搜索知识库”属于错误承诺。
- 模块页提供稳定章节 ID 和持续可达的章节目录。桌面端目录在滚动时保持可达并标示当前章节；移动端使用等价的章节条或抽屉，不隐藏任何主要章节。
- 页面默认先展示结论、机制、边界与售前判断，再渐进展开案例、失败诊断、证据细节和延伸内容。高风险边界、关键前提和 Go / No-Go 条件不得默认折叠。
- 章节内出现真实客户情境时，直接给出短答、下一问或指向对应问答的入口。跳转必须能定位到具体问题，并提供返回原章节的路径，不让读者在问答长列表中重新寻找。
- 章节链接、问题链接和来源链接应保持可复制、可分享；交互增强不得依赖 Login、个人状态或仅在某台设备保存的阅读记录。

## 4. 机制、因果与比较视图

- 流程、循环、状态变化、责任传递、层级和因果链优先使用能表达方向与关系的网页视图，不把每个节点退化成同权重的普通卡片。
- 机制视图应同时回答：输入是什么、发生了什么、输出或状态如何变化、哪里会失败、由谁控制、售前据此判断什么。
- 可交互视图只能切换真实且互斥的分析视角，例如离线 / 在线链、正常 / 故障路径、只读 / 写入风险或候选方案；切换后必须产生新的判断信息。
- 图、卡、表、案例和问答数量均由知识需要决定。没有必要的关系就不用图，没有共同维度就不用表，不能为了“网页看起来丰富”制造装饰组件。
- 表格只用于两个或更多对象在同一组维度上的真实比较。单对象说明、松散事实、步骤和长段解释改用正文、列表、机制图或诊断路径。
- 桌面表格需要清晰表头和行列关系；移动端优先把每行转成带字段名称的语义卡片。确需横向滚动时必须提供可见提示，并保证页面本身没有全局水平溢出。

## 5. 动态均衡卡片

- 卡片数量来自实际内容，不把 8 张或其他数量写成模板前提。
- 桌面端按当前数量自动计算完整行，每行最多 4 张，各行数量差不超过 1。例如 5 张为 3+2、6 张为 3+3、7 张为 4+3、9 张为 3+3+3。
- 平板采用双列；单数时最后一张占满最后一行，避免孤立空位。手机采用单列。
- 不使用 `auto-fit` 任由浏览器产生 5+3、4+1 等残缺构图；布局算法必须覆盖单数、双数、空列表与未来增减。
- 均衡不等于固定等高。除非信息比较需要对齐，不给内容卡设置大面积固定 `min-height`，不以空白换取 PPT 式整齐。

## 6. 字体、标题与换行

- 连续中文正文、问答、表格正文和机制解释在桌面、平板与手机均不低于 16px，桌面建议 17–18px；次级说明不低于 14px。
- 只有不承担知识理解责任的编号、眉题、来源等级或日期等元数据可以使用 12px；不得让 9–11px 字号承载定义、边界、判断、表格内容或客户回答。
- 正文建议采用 1.65–1.85 行高和约 42–48 个汉字的阅读宽度。标题字号服务层级，不以超大标题和大面积首屏空白模仿 PPT 封面。
- 中文字体应在支持平台上保持一致、清晰和可维护；中英混排需检查字重、基线和标点，不依赖系统随机回退获得视觉效果。

- 数字与量词、英文缩写与中文解释、专有名词不得被任意拆开。
- 复杂标题拆成“主标题 + 统计单元 / 副标题”，不依赖浏览器在任意字符处换行。
- 分隔符必须与后一语义单元绑定，换行时不得单独留在上一行或下一行。
- 使用 `white-space: nowrap` 保护短语，只在设计好的单元边界换行。
- 标题在常见桌面宽度下不应出现只有 2–4 个汉字的孤行。
- 粘性页头和章节导航存在时，为每个锚点预留正确滚动偏移，跳转后标题不能被遮挡。

## 7. 模块复用与 PPT 参考边界

每个模块用以下骨架检查知识完整性，并按主题需要增减、合并或调整顺序：

1. 模块定位与相关模块
2. 基础概念与核心原理
3. 机制、架构与变体
4. 适用边界与选型
5. 云服务连接与方案组合
6. PoC、评估、安全与运营
7. 客户问题实战包
8. 来源、证据等级与复核日期的 Reference 回链

结构是质量检查清单，不是数量配额。没有价值的图、卡片或说明不应为了对齐模板而添加。

来自 `external_reference/` 的讲义、白皮书或演示材料可以帮助识别概念、架构关系、客户问题和云服务连接，但不自动成为公开证据。动态事实、性能数字、协议版本、价格、法规与产品能力必须回到论文、标准或官方文档核验，再以稳定 `sourceId` 进入 Reference 台账。

PPT 只作为内容地基和问题线索：

- 不沿用其目录作为网页目录，不以“逐页覆盖”作为内容完成标准；
- 不复制其卡片数量、固定比例、超大标题、小字注释或一页一结论的节奏；
- 网页必须补充 PPT 难以提供的检索、章节定位、渐进解释、机制切换、情境问答回链和内联证据体验；
- 若页面只是把 PPT 文本换成卡片或表格纵向排列，应判定为未达到 HTML 版本标准。

## 8. 问答与证据同步

- 模块正文、证据卡、客户问答和来源台账共用稳定的 `sourceId`；来源标题、链接、证据类型和核验日期只在 `app/reference-content.mjs` 维护，并统一呈现在 `/references`。
- 每道客户问题必须包含结论短答、深一层、售前下一问和“本题依据”。“本题依据”不仅列来源，还要写明该来源具体支持什么、不能支持什么。
- 论文实验、厂商实验、官方框架和社区指南必须区分。来源等级代表证据类型，不表示结论可以跨数据集、客户或产品版本直接外推。
- 时延、成本、SLA、TCO、地域和产品能力等客户相关结论，应显式标注需以当期产品资料、客户基线或 PoC 实测为准。
- 正文知识发生变化时，同一次变更必须检查相关问题、证据卡和来源台账；缺少有效来源的问答不得发布为确定事实。
- 问答总区可使用搜索与标签筛选，但核心章节仍应显示相关情境问题或明确入口；问答筛选后必须保留稳定问题锚点、当前结果数量和清除筛选路径。
- 点击证据可先展示“支持什么、不能外推什么、证据类别和核验日期”，同时保留 `/references#source-<sourceId>` 作为稳定完整台账路径。

## 9. 交互与可访问性

- 每个搜索、筛选、标签、章节切换、展开和快捷键都必须帮助读者定位、比较、理解或判断；纯装饰互动不得与知识控件争夺注意力。
- 控件显示的能力必须真实实现，例如显示 `⌘K` 就必须能聚焦搜索；显示筛选数量就必须与当前结果一致。
- 键盘、触控和屏幕阅读器均能到达全部功能；不能依赖 hover 才显示关键解释。焦点样式、当前章节、选中标签和展开状态必须可见。
- Tabs、Dialog、Accordion 等模式遵守相应 ARIA 关系和键盘行为；禁用动效或 `prefers-reduced-motion` 时仍可完整使用。
- JavaScript 失败时，正文、章节锚点、客户问答和 Reference 链路仍保持可阅读；交互是渐进增强，不是内容存在的前提。

## 10. 云服务事实呈现

- 云服务首先写真实能力、客户价值和责任边界，不先堆厂商产品名。
- 一旦点名具体产品或版本，必须同时说明官方生命周期状态、适用地域、已知限制、配额 / SLA（若相关）、计价单位、核验日期和稳定 `sourceId`。
- 明确云平台、客户应用、数据与身份系统、人工审批各自负责什么；托管能力不能被写成业务正确性、合规或客户效果保证。
- Preview、地域、价格、配额、模型 / 硬件型号等快速变化事实必须按采购时点重新核验，不能只在页脚写一个统一日期。

## 11. 桌面与移动端视觉 QA

- 自动布局测试只证明结构未明显破坏，不能替代真实页面视觉复核。
- 每次影响共享布局、字体、导航、表格、图解或交互时，至少检查代表性桌面、平板和手机宽度；优先覆盖约 1440px、1024px 和 390px 视口，也可使用能暴露同类问题的等价尺寸。
- 使用真实最长标题、最长中英术语、最大字段表格、单数 / 双数卡片、展开问答、筛选无结果和重要边界状态检查页面。
- 人工复核至少检查：首屏信息密度、正文与次级字号、中文换行、标题遮挡、sticky 导航、焦点、触控目标、横向溢出、表格转换、展开 / 返回路径和来源上下文。
- 视觉 QA 发现系统性问题时，同步更新共享组件、规范与可自动化检查，不为单个模块增加 CSS 特判。

## 12. 发布前检查

- 内容：标题是否为知识库语气；术语是否中英对应；结论是否有边界。
- 构图：内容是否形成明确机制或判断层级；动态卡片是否均衡；表格是否真的用于比较；标题是否按完整语义单元换行。
- 字体：正文、问答、表格和次级说明是否达到字号下限；是否存在用小字隐藏高密度内容的情况。
- 响应式：桌面、平板、手机的列数、阅读顺序、表格转换、展开状态和触控路径是否清晰。
- 导航：首页、全局搜索、sticky 章节目录、情境问答、相关模块、Reference 回链和页内锚点是否有效。
- 交互：每个搜索、筛选、快捷键和切换是否真实可用、可访问，并能帮助判断而非装饰。
- 云服务：具体产品是否包含能力、状态、地域、限制、计价单位、核验日期与责任边界。
- 视觉 QA：是否使用真实内容完成桌面与移动端人工复核，而不是只看构建和自动测试通过。
- 维护：来源、核验日期、相关模块和题内依据是否可追踪，所有 `sourceId` 是否能解析到唯一来源。

内部事实台账、复核节奏与发布门槛见 `CONTENT-MAINTENANCE.md`，不得作为知识章节展示在读者页面。
