# 新模块质量门禁与历史问题追溯矩阵（内部）

本文把已出现过的内容、构图、导航、Portable、证据与发布问题转成后续模块必须通过的永久门禁。它只用于编辑、评审和自动检查，不能作为读者页面的章节、卡片、页脚、提示语或构建说明。

本文件回答“过去哪里出过问题、以后由什么机制阻止复发”；`MODULE-BUILD-STANDARD.md` 回答“一个好模块应该怎样建设”。两份文件必须同时执行。

## 1. 门禁分层

每条要求必须落在以下至少一层，不能只停留在口头约定：

1. **统一数据源**：模块发布状态、术语、来源和内容契约只维护一份；页面与测试从它派生。
2. **共享组件**：标题、重要边界、动态卡片等高风险视觉使用共享实现，不复制模块特例。
3. **自动检查**：检查路由、结构、术语、证据、时效、匿名访问和任意数量布局。
4. **人工内容评审**：判断原理是否真正讲清、译法是否专业、云服务连接是否成立、回答是否能推进客户对话。关键词测试不能替代这一层。
5. **人工页面复核**：使用真实内容检查桌面、平板和移动端的排版、字号、导航、机制视图、交互、表格转换、焦点与返回路径。构建和结构测试不能替代这一层。
6. **发布证据**：构建、检查、推送和公开站点必须对应同一个 Git 提交。

## 2. 历史问题 → 永久门禁

| 类别 | 曾出现的症状 | 永久规则 | 工程控制 | 人工复核 |
| --- | --- | --- | --- | --- |
| 原理深度 | RAG 有流程图和名词，但基础原理不足 | 核心主题必须按“定义—输入输出—机制—失败—控制—售前判断—证据”讲清；删掉无用公式不能同时删掉直观原理 | 已发布模块在 `app/module-publication.mjs` 登记 `contentContract.principle` 与 `mechanism`，生产 HTML 必须出现对应语义和标准原理区块 | 读者能否不用背术语解释为什么有效、为什么失败 |
| 动作循环 | Agent 未逐项解释感知、思考、行动、观察 | 行业通行生命周期和动作循环逐项讲职责、前后输入输出、失败与停止 | 模块契约登记不可丢失动作；页面使用原理区块和中英术语 | 不能把关键词出现当成讲清 |
| 核心组件 | Agent 未深入规划、记忆、工具 | 关键组件独立解释工作方式、协作关系、数据责任、风险、云落点和验收 | 模块契约登记机制语义；统一术语表校验中英名称 | 判断是否遗漏该主题公认的核心组件 |
| 理论标尺 | 内容在“过浅”与“数学过深”之间摇摆 | 理论深度以是否改变架构、选型、风险或验收判断为门槛；无不可替代售前价值的公式不展示 | 模块公式策略与禁用公式检查；Reference 可保留论文来源 | 对每个公式回答“会让售前作出什么不同判断” |
| 数据支撑 | 数字被写成普遍承诺 | 数字绑定数据集、模型/产品版本、条件、日期、来源和不可外推边界 | 证据卡逐字段校验；来源必须在统一台账且未超过复核周期 | 判断实验是否能支持正文的措辞强度 |
| 问答同步与覆盖 | 正文变了，高频问题和深答仍是旧结论；课程扩充后机械地每章配题，或把不同模块补成相同题数 | 正文重要边界、云机会、常见反对意见变化时，同步复核问答、证据卡和 Reference；只有能够改变方案选择、风险判断、验收方式或下一步行动的问题才独立成题。相近问题合并为一个完整回答，不要求逐章配题，也不追求模块间题数一致 | 已发布模块必须有问答；重点模块在发布契约登记必须覆盖的问题主题并逐项检查，不使用固定题数门槛；题内依据逐项解析，来源归入本模块分组；自动审计补充题数和最终题数是否异常收敛为统一模板，并验证新增主题全部进入发布契约 | 逐题说明其不可替代的客户决策价值；深答是否增加机制、权衡和实施含义，下一问是否推进发现或 PoC；问题之间不得只是换词重复，删去一题后若没有损失独立判断，应合并或删除 |
| PPT 变网页 | HTML 只是把 PPT 的目录、卡片和小字纵向搬运，或只借了几处标题而遗漏大部分知识 | PPT 只作为内容地基与查漏线索；建设前先形成知识点、关系、案例、风险和练习的覆盖清单，网页按客户判断链重组，并在机制、应用、验证和时效上扩展 | 内部规范禁止沿用 PPT 目录与固定构图；有本地讲义时，自动检查章节和学习路线达到模块登记的最低深度，正式模块登记稳定章节与问题锚点 | 对照讲义逐项标记“吸收并重组 / 扩展 / 有理由省略”，同一知识在 HTML 中是否更快定位、形成心智模型并复用 |
| 学习闭环 | 模块有很多知识卡，但读者不知道知识全貌、先学什么、学完能做什么，也没有可验证练习；路线已扩展却仍显示“固定三步”等旧标题 | 每个共享模块提供可观察学习结果、按知识复杂度决定的推荐路线、带机制/判断/边界的课程地图，以及带产物/通过标准的实战任务；外部材料只提供覆盖线索；读者标题不编码固定步骤数 | `module-learning-content.mjs` 与 `module-curriculum-content.mjs` 覆盖全部共享模块；路线允许 3–6 步，标题使用共享的“建议学习顺序”，所有题内来源解析并进入当前模块 Reference；公开页不得出现改写过程 | 读者能否先形成完整知识版图，再走到方案判断并交付一份可评审的真实产物；路线数量是否由问题决定而非套模板；页面标题与实际路线是否一致 |
| 可检索入口 | 模块增多后只能横向滑卡片或逐层翻目录 | 全局检索至少覆盖模块、术语、章节、客户问题和实战练习，结果链接到稳定地址；搜索范围和界面文案必须一致 | 搜索索引从发布注册表、术语和内容数据派生；自动检查代表性查询与链接 | 售前能否从一个客户关键词在少量操作内进入正确模块、具体问题或可执行练习 |
| 长文导航 | RAG / Agent / Prompt 有十多个全展开小节，但页头只放少数锚点，滚动后失去位置 | 每个正式模块提供完整章节目录；桌面 sticky 并标示当前章节，移动端提供等价入口 | 章节 ID 来自共享内容结构；检查目录覆盖、锚点、当前状态和粘性页头滚动偏移 | 从任意章节能否知道所在位置、跳到目标并返回原阅读路径 |
| 渐进阅读 | 结论、细节、案例、证据和边界同权重全部铺开，形成超长信息墙 | 先呈现结论、机制、边界和判断，再按需展开细节；重要边界与 Go / No-Go 不默认折叠 | 共享展开组件保留语义、锚点和无脚本基本内容；检查关键边界不在默认隐藏区 | 快速浏览能否得到完整判断，精读又能否继续获得因果和证据 |
| 封面式首屏 | 模块首屏被超大标题、大色块和装饰留白占满，第一条真实判断落到下一屏；内容总量很大但入口密度很低 | 模块首屏必须同时建立主题、定义、内容规模和阅读入口；标题不以封面尺寸挤压模块概览，首个正文区优先呈现可行动结论 | 视觉配置从 `module-publication.mjs` 派生；高密度阅读配置使用共享 Hero 概览和阅读壳，测试所有启用该配置的模块均输出章节、客户问题和证据概览 | 在 1366×768、1440×900 首屏检查主题、定义、内容概览与首条判断是否形成连续阅读起点，而不是只看到封面 |
| 机制与因果视图 | 流程、状态和责任关系被拆成一组普通卡片，难以形成心智模型 | 核心流程、循环、状态、层级和因果关系使用适合的方向性视图；视图同时说明输入、变化、输出、失败、控制和判断 | 机制视图使用共享语义组件并保持内容驱动；不为模块写固定节点数量 CSS | 读者能否解释“为什么有效、哪里失败、谁负责”，而不是只复述节点名称 |
| 情境问答回链 | 原理与客户问答相隔很远，读者需在长列表重新寻找 | 重要章节直接呈现相关情境问题，或链接到具体问答并支持返回原章节 | 问题有稳定 ID、章节关系与返回参数 / 锚点；筛选不破坏链接 | 学完一个机制后能否立即把它转成客户短答、深答和下一问 |
| 按需呈现 | 为放图、卡片或满足数量而堆内容 | 图、卡、表、案例和问答均不设数量配额；只在比正文更清楚时使用 | 证据卡允许为空；共享 `BalancedGrid` 与 `balanceGridRows` 支持 0、1、单数、双数和未来增减 | 删除该视觉后是否反而更清楚 |
| 术语中英 | 检索、增强、生成等专业词缺英文 | 专业术语首次出现给出准确中英对照；同一术语全站保持同一译法 | `app/terminology.mjs` 是术语注册表；模块发布契约引用稳定 `termId` | 由懂领域的人检查译法，不直接发布逐词机翻 |
| 缩写理解 | RAG、HNSW、TTFT、IAM 等缩写在高密度页面反复出现，读者必须离开当前上下文另查全称 | 高频缩写在首次阅读入口提供全称与一句话说明；桌面可悬停 / 聚焦，触屏可点击，正文定义不依赖提示气泡 | 缩写、全称和说明统一登记在 `terminology.mjs`；共享原生 `details` 组件输出可访问名称并在首页和目标模块按需使用 | 检查提示是否真正消除理解阻塞，而不是给每个英文词增加装饰性交互 |
| 普通中文 | `客户信号`、`来源台账`、`知识供应链`、`责任链`、`运营闭环` 等内部或咨询式简称进入读者页面；`控制面`、`终态`、`基线`、`映射` 脱离技术语境反复出现 | 标题、标签和解释优先使用具体名词与日常动词；必要技术词首次出现先用普通中文解释，仅在技术区别重要时保留英文或缩写 | 全部公开路由检查高置信禁用表达；术语注册与模块标准固定首次解释规则 | 逐段朗读，判断有 Python / API 基础的读者能否一次读懂；检查是否反复套用同一写作句式 |
| 未来双语 | 希望未来有英文版，但不能牺牲准确性 | 中英文共享概念 ID、结构和来源关系，英文文案独立撰写、独立校对 | 当前先校验术语 ID；英文版上线时校验两种语言的概念与来源集合 | 独立英文技术审校 |
| 知识库语气 | “先确定”等像编辑者对读者发指令 | 标题用客观陈述或名词短语；流程动词只描述系统或客户动作 | 公开标题禁用词检查 | 判断标题是否像长期知识资产而非制作对话 |
| 内部说明 | BUILD BRIEF、读者画像、语言策略、维护规则等被展示 | 编辑、语言、构建、维护和发布信息不进入公开页面 | 全部公开路由执行黑名单；页面不导入内部规范 | 页面上的每块信息是否帮助学习或客户交流 |
| 层级说明 | 每层泛化说明空泛且挤占模块空间 | 首页知识地图只展示层名和可导航模块；不显示 `layerPurpose` | 首页 HTML 明确禁止渲染层级 purpose；通用模块页也不继承显示 | 模块卡是否获得足够阅读空间 |
| 重要边界 | 重要限制与普通说明视觉权重相同 | 高影响边界统一用高对比、可访问的共享组件 | `CriticalBoundary` 统一输出 `data-importance="critical"`；每个已发布模块至少一处 | 该边界是否真的会导致过度承诺、风险或错误架构 |
| 字号与遮挡 | 正文字号偏小；大量 9–14px 文字承载知识；Agent 标题和副标题重叠 | 连续正文、问答、表格正文和机制解释不低于 16px，次级说明不低于 14px；标题用真实几何间距且不以超大封面挤压正文 | 共享排版 token 与 `.moduleHeroTitle`；样式检查字号下限并禁止模块 ID 特判 | 桌面、平板、手机检查字级、行宽、重叠、溢出和中英混排 |
| 证据卡层级 | “可引用事实”用接近封面标题的数字或短语占满卡片，结论、边界和来源被压成小字 | 证据卡按事实标签、结论、适用边界、来源连续阅读；标签只负责定位，不压过结论，卡片不靠固定大高度对齐 | `ModuleEvidenceGrid` 统一结构；事实标签设置紧凑上限，结论与边界满足 16px / 14px 下限，移除强制大高度 | 在真实长短证据上检查读者第一眼能否读到“这说明什么”和“不能外推到哪里” |
| 孤行与语义换行 | “28 个细 / 分模块”等数字量词被拆开 | 数字量词、缩写释义和专有名词作为完整语义单元 | 统计词组使用独立内联单元和不拆分样式 | 常用宽度检查是否出现 2–4 字无意义孤行 |
| 动态构图 | 8 张卡片出现 5+3；漏传 span 后卡片压成窄列；5 张卡产生非法 `span 2.4` | 桌面均衡分行，行差不超过 1；按正文容器宽度降为双列；手机单列；变量失效时通栏保底 | `balanceGridRows` 与 `gridSpan` 属性测试覆盖任意数量并禁止分数跨度；所有 `--*-span` 必须有通栏 fallback；真实浏览器检查行宽覆盖、极窄列、文字逃逸和重叠 | 不因追求对称删减或凑出内容；左侧目录存在时仍按实际正文宽度验收 |
| PPT 式等高 | 大量固定 `min-height` 用空白强行对齐，长短内容失去自然节奏 | 动态布局只平衡行，不强制内容卡大面积等高；只有真实比较需要时才对齐 | 共享卡片默认 `min-height: auto`；检查高风险固定高度选择器 | 删除固定高度后是否更易连续阅读，比较关系是否仍清楚 |
| 表格滥用 | 单对象说明和长段文字被塞进表格；手机只能横向拖动 780–900px 内容 | 表格只用于两个以上对象在共同维度上的比较；手机优先转语义卡片，确需横滚时给出提示 | 共享 ComparisonMatrix 支持桌面表格 / 移动卡片；检查全局水平溢出 | 不看列标题能否理解每个移动条目，表格是否真的帮助比较 |
| 装饰性交互 | 搜索只搜少量卡片却写“搜索知识库”；显示快捷键但未实现；动画不增加判断 | 每个交互必须帮助定位、比较、展开、追证或决策；界面承诺与实际范围一致，关键内容不依赖 hover | 交互测试覆盖搜索范围、快捷键、筛选结果、键盘、触控和无脚本回退 | 关闭交互后是否仍能阅读；使用交互后是否获得新的判断信息 |
| 视觉一致 | 05、06、07 缺少彩色条 | 同类层级必须都取得 accent，且新增层也不能掉色 | 色条选择器按周期而非固定 1–7；测试每层都有规则 | 颜色只表达层级，不误导发布状态 |
| 首页偏置 | 已有多个模块，首页仍固定突出 RAG | 首页入口从发布注册表生成；主 CTA 指向模块选择或知识地图，不指向单一模块 | `publishedModules` 是唯一发布状态源；首页和动态路由从它派生 | 新模块发布后入口排序是否符合学习逻辑 |
| 导航串联 | RAG 标签只是装饰；跨模块概念没有回链 | 所有模块名和相关模块都使用稳定可达 URL；局部解释足以完成当前阅读 | 路由 200、无自链接、统一说“相关模块” | 关系是否真实、是否遗漏关键概念，不为对称硬加关系 |
| 云服务连接 | 技术内容与云销售目标断开、只列产品名，或把“支持”写成没有状态和边界的承诺 | 对真实相关环节写“技术环节—真实能力—客户价值—发现问题—验收—责任边界”；点名具体产品时同时写生命周期状态、地域、限制、计价单位，并通过来源显示核验日期 | 内容契约要求云区块；快速变化产品字段用 `sourceId` 与时效策略校验，从统一台账解析 `verifiedAt`，超期阻止发布 | 区分协议、模型、单一产品与完整平台能力；确认云平台与客户应用、数据、身份、人工各自责任 |
| 独立页面 | 多个模块正文堆回首页，不易维护分享 | 首页只保留入口、知识地图和学习路径；每个模块有稳定独立路由 | 发布注册表记录稳定 path；所有知识地图路由匿名返回 200 | 链接能否直接发给同事并保留语义上下文 |
| Reference | 来源散落主页面，维护重复 | `/references` 是唯一公开来源台账；模块只用 `sourceId` 回链 | 外部 URL 只能存在于 `app/reference-content.mjs`；分组和锚点唯一 | `supports` 与边界是否准确，不把来源名称当证明 |
| 时效性 | 只有核验日期，没有真正过期门槛 | 按事实变化速度使用 30/90/180 天最长复核间隔；超期来源阻止发布 | `sourceFreshness` 严格解析日期、拒绝未来日期和超期来源 | 产品公告、安全事件或客户反证可提前触发复核 |
| Portable | 曾有 Login；源文件不在指定仓库；担心包大小导致删内容 | 知识页匿名直达；仓库是唯一源文件事实源；不依赖个人会话和本机路径；内容优先，不以包大小删知识 | 全路由匿名 GET；无登录表单/密码框/本机路径；干净构建；源文件全部在仓库 | 从全新环境能否构建、模块 URL 能否直接分享 |
| 多载体 | 未来希望 HTML、PPT、离线版本一致 | 事实、术语、边界与来源从同一内容源派生；载体只改变呈现 | 当前先统一内容数据、术语和来源 ID；后续生成器比较概念覆盖 | 载体压缩不能静默删掉关键前提 |
| 响应式视觉 QA | 自动测试通过，但真实页面仍出现小字、中文断行、粘性遮挡、表格难读或触控问题 | 每次共享视觉、导航或交互变化必须用真实内容完成笔记本优先、平板和移动端人工复核 | 优先检查 1366×768 与 1440×900，并保留最长标题、展开问答、表格和无结果状态的 QA 清单；自动测试只做辅助 | 检查首屏知识可见度、侧栏占宽、字号、换行、焦点、触控、横向溢出、返回路径和信息层级 |
| 首次打开空白 | Chrome 直接进入首页时只收到标题与样式，因中文 Web Font 大量分片被全量预加载而长期空白、持续转圈 | 首屏不得注册或预加载整套大字符集中文 Web Font；优先使用本机中文字体栈；正式地址必须用全新 Chrome 直接导航验收 | 根布局不通过 `next/font` 注册大字符集中文字体；自动检查系统字体栈存在且响应 `Link` 头不含中文字体分片；发布后检查首页从空标签到 `document.readyState=complete` 且正文存在 | 用未预热缓存的新标签分别直达首页和代表模块，确认标题、首屏正文和交互都能完成加载 |
| 推送与公开页 | GitHub 已更新但公开页面仍旧 | 任何推送后必须用同一精确提交创建 Sites 版本并公开部署，成功后才算完成 | 发布时核对本地 SHA、远端 SHA、Sites 源提交和部署状态 | 打开公开 URL 确认交付地址 |

## 3. 新模块发布登记顺序

新主题可以先进入知识地图，但只有完成以下全部步骤后才能加入 `publishedModules`：

1. 建立稳定 `/modules/<slug>` 正式页面，首页不复制模块正文。
2. 在 `app/terminology.mjs` 登记本模块专业术语，并由专业人员确认中英名称。
3. 在 `app/reference-content.mjs` 建立唯一 Reference 分组和已核验来源。
4. 完成基础原理、行业通行动作/组件、关键边界、相关模块、云服务、PoC、问答和题内依据。
5. 在 `app/module-publication.mjs` 登记 path、标题 ID、术语 ID 与内容契约。
6. 使用共享阅读壳、检索入口、sticky 章节导航、渐进展开、机制 / 因果视图、`moduleHeroTitle`、`CriticalBoundary`、`BalancedGrid`、`balanceGridRows` 和 `gridSpan`；不得复制模块 ID CSS 补丁。
7. 为核心章节建立情境问答入口或可返回的具体问题链接；点名云产品时补齐状态、地域、限制、计价单位、核验日期与责任边界。
8. 通过构建、代码检查、全路由匿名访问、来源解析、时效、任意数量布局、交互真实性与内部信息隔离测试。
9. 完成人工内容评审与桌面 / 平板 / 移动端视觉 QA，确认术语、原理深度、云服务价值、客户话术和网页阅读体验，而不是只看测试通过。

## 4. 新模块 Definition of Done

### 内容

- [ ] 模块定位明确；基础概念、输入输出、核心机制、行业通行动作/组件和失败条件讲清。
- [ ] 原理能支持售前判断，没有用术语堆叠或无用公式冒充深度。
- [ ] 专业术语在统一术语表登记，首次出现有准确中英对照。
- [ ] 标题、标签和解释使用普通中文；没有出现高置信内部 / 咨询式简称，必要技术词首次出现有直白解释。
- [ ] 重要边界醒目，且说明模型、应用、身份、数据或人工各自责任。
- [ ] 真实相关的云能力连接到客户价值、发现问题和验收；没有机械填满所有云产品类别。
- [ ] 高频问题、深答、下一问、题内依据与正文一致；每道题都有独立决策价值，相近问题已经合并，没有按章节配额或为了与其他模块题数对齐而出题。

### 页面与 Portable

- [ ] 页面只有一个主标题；标题、副标题和定义在常用宽度不遮挡。
- [ ] 全局检索可从模块、术语、章节、客户问题或实战练习进入稳定地址，搜索范围与界面文案一致。
- [ ] 章节目录覆盖主要内容；桌面 sticky 并标示当前章节，移动端提供等价入口，锚点不被粘性页头遮挡。
- [ ] 页面先展示结论与判断，再渐进展开细节；重要边界不默认折叠，章节情境问答可定位并返回。
- [ ] 核心流程、状态、层级或因果关系使用适合视图，交互能增加理解或判断而不是装饰。
- [ ] 连续正文、问答、表格正文和机制解释不低于 16px，次级说明不低于 14px。
- [ ] 高频缩写在首次阅读入口可悬停 / 聚焦和触屏点击查看全称与简短说明，关键定义不依赖 hover 才能获得。
- [ ] 证据卡优先呈现结论和适用边界，事实标签没有使用封面级字号，卡片高度随内容自然生长。
- [ ] 图、卡、表和问答按需使用；0、1、单数、双数和未来增减均不破版。
- [ ] 表格只用于共同维度的真实比较，移动端可在不依赖隐蔽横向滚动的情况下理解主要结论。
- [ ] 首页只增加入口；模块正文、完整来源和内部说明没有回流首页。
- [ ] 页面匿名可读、稳定 URL 可直接分享；无 Login、个人会话、本机路径或仓库外必需源文件。
- [ ] 相关模块可导航，术语和边界与主要归属模块一致。
- [ ] 使用真实内容完成桌面、平板和移动端人工视觉 QA，覆盖长标题、展开问答、表格、sticky 导航、焦点和返回路径。
- [ ] 使用未预热缓存的 Chrome 标签直接打开首页与代表模块，页面能进入完成状态并显示正文；响应 `Link` 头不含大字符集中文字体分片。

### 证据与发布

- [ ] 所有 `sourceId` 可解析、已分组、未超复核周期；数字写明条件与不可外推边界。
- [ ] 具体云产品事实包含真实能力、生命周期状态、地域、限制、计价单位、核验日期、责任边界和稳定来源。
- [ ] 正文变化已同步检查问答、证据卡、相关模块和 Reference。
- [ ] 构建、自动检查和代码检查通过；人工内容评审已完成。
- [ ] 若发生 Git 推送，公开 Sites 已部署同一精确提交并成功。

## 5. 新反馈的处理规则

以后发现新的系统性问题，修复当前页面只是第一步；同一次变更还必须：

1. 在本追溯矩阵增加或更新对应问题；
2. 更新建设标准或共享组件；
3. 能自动判断的部分增加通用测试，不能自动判断的部分写入人工复核；
4. 检查其他已发布模块是否存在同类问题；
5. 在发布前验证公开页面使用的是修复后的精确提交。

不得通过为单个模块增加特殊选择器、固定数量分支或特定句子断言来假装问题已经系统解决。
