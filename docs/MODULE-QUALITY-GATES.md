# 新模块质量门禁与历史问题追溯矩阵（内部）

本文把已出现过的内容、构图、导航、Portable、证据与发布问题转成后续模块必须通过的永久门禁。它只用于编辑、评审和自动检查，不能作为读者页面的章节、卡片、页脚、提示语或构建说明。

本文件回答“过去哪里出过问题、以后由什么机制阻止复发”；`MODULE-BUILD-STANDARD.md` 回答“一个好模块应该怎样建设”。两份文件必须同时执行。

## 1. 门禁分层

每条要求必须落在以下至少一层，不能只停留在口头约定：

1. **统一数据源**：模块发布状态、术语、来源和内容契约只维护一份；页面与测试从它派生。
2. **共享组件**：标题、重要边界、动态卡片等高风险视觉使用共享实现，不复制模块特例。
3. **自动检查**：检查路由、结构、术语、证据、时效、匿名访问和任意数量布局。
4. **人工内容评审**：判断原理是否真正讲清、译法是否专业、云服务连接是否成立、回答是否能推进客户对话。关键词测试不能替代这一层。
5. **发布证据**：构建、检查、推送和公开站点必须对应同一个 Git 提交。

## 2. 历史问题 → 永久门禁

| 类别 | 曾出现的症状 | 永久规则 | 工程控制 | 人工复核 |
| --- | --- | --- | --- | --- |
| 原理深度 | RAG 有流程图和名词，但基础原理不足 | 核心主题必须按“定义—输入输出—机制—失败—控制—售前判断—证据”讲清；删掉无用公式不能同时删掉直观原理 | 已发布模块在 `app/module-publication.mjs` 登记 `contentContract.principle` 与 `mechanism`，生产 HTML 必须出现对应语义和标准原理区块 | 读者能否不用背术语解释为什么有效、为什么失败 |
| 动作循环 | Agent 未逐项解释感知、思考、行动、观察 | 行业通行生命周期和动作循环逐项讲职责、前后输入输出、失败与停止 | 模块契约登记不可丢失动作；页面使用原理区块和中英术语 | 不能把关键词出现当成讲清 |
| 核心组件 | Agent 未深入规划、记忆、工具 | 关键组件独立解释工作方式、协作关系、数据责任、风险、云落点和验收 | 模块契约登记机制语义；统一术语表校验中英名称 | 判断是否遗漏该主题公认的核心组件 |
| 理论标尺 | 内容在“过浅”与“数学过深”之间摇摆 | 理论深度以是否改变架构、选型、风险或验收判断为门槛；无不可替代售前价值的公式不展示 | 模块公式策略与禁用公式检查；Reference 可保留论文来源 | 对每个公式回答“会让售前作出什么不同判断” |
| 数据支撑 | 数字被写成普遍承诺 | 数字绑定数据集、模型/产品版本、条件、日期、来源和不可外推边界 | 证据卡逐字段校验；来源必须在统一台账且未超过复核周期 | 判断实验是否能支持正文的措辞强度 |
| 问答同步 | 正文变了，高频问题和深答仍是旧结论 | 正文重要边界、云机会、常见反对意见变化时，同步复核问答、证据卡和 Reference | 已发布模块必须有问答；题内依据逐项解析，来源归入本模块分组 | 深答是否增加机制/权衡/实施含义，下一问是否推进发现或 PoC |
| 按需呈现 | 为放图、卡片或满足数量而堆内容 | 图、卡、表、案例和问答均不设数量配额；只在比正文更清楚时使用 | 证据卡允许为空；共享 `BalancedGrid` 与 `balanceRows` 支持 0、1、单数、双数和未来增减 | 删除该视觉后是否反而更清楚 |
| 术语中英 | 检索、增强、生成等专业词缺英文 | 专业术语首次出现给出准确中英对照；同一术语全站保持同一译法 | `app/terminology.mjs` 是术语注册表；模块发布契约引用稳定 `termId` | 由懂领域的人检查译法，不直接发布逐词机翻 |
| 未来双语 | 希望未来有英文版，但不能牺牲准确性 | 中英文共享概念 ID、结构和来源关系，英文文案独立撰写、独立校对 | 当前先校验术语 ID；英文版上线时校验两种语言的概念与来源集合 | 独立英文技术审校 |
| 知识库语气 | “先确定”等像编辑者对读者发指令 | 标题用客观陈述或名词短语；流程动词只描述系统或客户动作 | 公开标题禁用词检查 | 判断标题是否像长期知识资产而非制作对话 |
| 内部说明 | BUILD BRIEF、读者画像、语言策略、维护规则等被展示 | 编辑、语言、构建、维护和发布信息不进入公开页面 | 全部公开路由执行黑名单；页面不导入内部规范 | 页面上的每块信息是否帮助学习或客户交流 |
| 层级说明 | 每层泛化说明空泛且挤占模块空间 | 首页知识地图只展示层名和可导航模块；不显示 `layerPurpose` | 首页 HTML 明确禁止渲染层级 purpose；通用模块页也不继承显示 | 模块卡是否获得足够阅读空间 |
| 重要边界 | 重要限制与普通说明视觉权重相同 | 高影响边界统一用高对比、可访问的共享组件 | `CriticalBoundary` 统一输出 `data-importance="critical"`；每个已发布模块至少一处 | 该边界是否真的会导致过度承诺、风险或错误架构 |
| 字号与遮挡 | 正文字号偏小；Agent 标题和副标题重叠 | 阅读型页面优先字号、行高和真实几何间距；标题只通过共享变量调整，不用模块 ID 补丁 | `.moduleHeroTitle`、CSS 变量与单一 H1；禁止 `#agent-title` 等特判 | 桌面、平板、手机检查无重叠、溢出和难读小字 |
| 孤行与语义换行 | “28 个细 / 分模块”等数字量词被拆开 | 数字量词、缩写释义和专有名词作为完整语义单元 | 统计词组使用独立内联单元和不拆分样式 | 常用宽度检查是否出现 2–4 字无意义孤行 |
| 动态构图 | 8 张卡片出现 5+3；固定 4×2 无法适应未来数量 | 桌面均衡分行，行差不超过 1；平板双列且单数尾项通栏；手机单列 | `balanceRows` 属性测试覆盖任意数量；共享 `BalancedGrid` 禁止 `auto-fit` 尾行漂移 | 不因追求对称删减或凑出内容 |
| 视觉一致 | 05、06、07 缺少彩色条 | 同类层级必须都取得 accent，且新增层也不能掉色 | 色条选择器按周期而非固定 1–7；测试每层都有规则 | 颜色只表达层级，不误导发布状态 |
| 首页偏置 | 已有多个模块，首页仍固定突出 RAG | 首页入口从发布注册表生成；主 CTA 指向模块选择或知识地图，不指向单一模块 | `publishedModules` 是唯一发布状态源；首页和动态路由从它派生 | 新模块发布后入口排序是否符合学习逻辑 |
| 导航串联 | RAG 标签只是装饰；跨模块概念没有回链 | 所有模块名和相关模块都使用稳定可达 URL；局部解释足以完成当前阅读 | 路由 200、无自链接、统一说“相关模块” | 关系是否真实、是否遗漏关键概念，不为对称硬加关系 |
| 云服务连接 | 技术内容与云销售目标断开，或只列产品名 | 对真实相关环节写“技术环节—云能力—客户价值—发现问题—验收方式” | 内容契约要求云区块；产品可变事实必须有来源与核验日 | 区分协议、模型、单一产品与完整平台能力 |
| 独立页面 | 多个模块正文堆回首页，不易维护分享 | 首页只保留入口、知识地图和学习路径；每个模块有稳定独立路由 | 发布注册表记录稳定 path；所有知识地图路由匿名返回 200 | 链接能否直接发给同事并保留语义上下文 |
| Reference | 来源散落主页面，维护重复 | `/references` 是唯一公开来源台账；模块只用 `sourceId` 回链 | 外部 URL 只能存在于 `app/reference-content.mjs`；分组和锚点唯一 | `supports` 与边界是否准确，不把来源名称当证明 |
| 时效性 | 只有核验日期，没有真正过期门槛 | 按事实变化速度使用 30/90/180 天最长复核间隔；超期来源阻止发布 | `sourceFreshness` 严格解析日期、拒绝未来日期和超期来源 | 产品公告、安全事件或客户反证可提前触发复核 |
| Portable | 曾有 Login；源文件不在指定仓库；担心包大小导致删内容 | 知识页匿名直达；仓库是唯一源文件事实源；不依赖个人会话和本机路径；内容优先，不以包大小删知识 | 全路由匿名 GET；无登录表单/密码框/本机路径；干净构建；源文件全部在仓库 | 从全新环境能否构建、模块 URL 能否直接分享 |
| 多载体 | 未来希望 HTML、PPT、离线版本一致 | 事实、术语、边界与来源从同一内容源派生；载体只改变呈现 | 当前先统一内容数据、术语和来源 ID；后续生成器比较概念覆盖 | 载体压缩不能静默删掉关键前提 |
| 推送与公开页 | GitHub 已更新但公开页面仍旧 | 任何推送后必须用同一精确提交创建 Sites 版本并公开部署，成功后才算完成 | 发布时核对本地 SHA、远端 SHA、Sites 源提交和部署状态 | 打开公开 URL 确认交付地址 |

## 3. 新模块发布登记顺序

新主题可以先进入知识地图，但只有完成以下全部步骤后才能加入 `publishedModules`：

1. 建立稳定 `/modules/<slug>` 正式页面，首页不复制模块正文。
2. 在 `app/terminology.mjs` 登记本模块专业术语，并由专业人员确认中英名称。
3. 在 `app/reference-content.mjs` 建立唯一 Reference 分组和已核验来源。
4. 完成基础原理、行业通行动作/组件、关键边界、相关模块、云服务、PoC、问答和题内依据。
5. 在 `app/module-publication.mjs` 登记 path、标题 ID、术语 ID 与内容契约。
6. 使用共享 `moduleHeroTitle`、`CriticalBoundary`、`BalancedGrid` 和 `balanceRows`；不得复制模块 ID CSS 补丁。
7. 通过构建、代码检查、全路由匿名访问、来源解析、时效、任意数量布局与内部信息隔离测试。
8. 完成人工内容评审，确认术语、原理深度、云服务价值和客户话术，而不是只看测试通过。

## 4. 新模块 Definition of Done

### 内容

- [ ] 模块定位明确；基础概念、输入输出、核心机制、行业通行动作/组件和失败条件讲清。
- [ ] 原理能支持售前判断，没有用术语堆叠或无用公式冒充深度。
- [ ] 专业术语在统一术语表登记，首次出现有准确中英对照。
- [ ] 重要边界醒目，且说明模型、应用、身份、数据或人工各自责任。
- [ ] 真实相关的云能力连接到客户价值、发现问题和验收；没有机械填满所有云产品类别。
- [ ] 高频问题、深答、下一问、题内依据与正文一致。

### 页面与 Portable

- [ ] 页面只有一个主标题；标题、副标题和定义在常用宽度不遮挡。
- [ ] 图、卡、表和问答按需使用；0、1、单数、双数和未来增减均不破版。
- [ ] 首页只增加入口；模块正文、完整来源和内部说明没有回流首页。
- [ ] 页面匿名可读、稳定 URL 可直接分享；无 Login、个人会话、本机路径或仓库外必需源文件。
- [ ] 相关模块可导航，术语和边界与主要归属模块一致。

### 证据与发布

- [ ] 所有 `sourceId` 可解析、已分组、未超复核周期；数字写明条件与不可外推边界。
- [ ] 正文变化已同步检查问答、证据卡、相关模块和 Reference。
- [ ] 构建、自动检查和代码检查通过；人工内容评审已完成。
- [ ] 若发生 Git 推送，公开 Sites 已部署同一精确提交并成功。

## 5. 新反馈的处理规则

以后发现新的系统性问题，修复当前页面只是第一步；同一次变更还必须：

1. 在本追溯矩阵增加或更新对应问题；
2. 更新建设标准或共享组件；
3. 能自动判断的部分增加通用测试，不能自动判断的部分写入人工复核；
4. 检查其他已发布模块是否存在同类问题；
5. 在发布前验证公开页面使用的是修复后的精确提交。

不得通过为单个模块增加特殊选择器、固定数量分支或特定句子断言来假装问题已经系统解决。
