# 售前知识库内容时效与发布维护规范（内部）

> 本文是编辑、巡检与发布人员使用的内部治理规范，不进入读者页面，也不作为知识正文、说明卡片或导航入口展示。公开页面只呈现对读者有直接价值的知识内容、证据来源与必要的核验信息。

## A. 事实最小单元

凡是可能随时间变化、适用范围变化或被新结论替代的陈述，都按一条“事实记录（Claim Record）”维护。记录至少包含：结论、适用范围、来源、证据等级、核验日、下次复核日、负责人、当前状态与替代历史。

标准字段 schema 如下：

| 字段 | 含义 | 维护要求 |
| --- | --- | --- |
| `claim_id` | 事实记录的稳定唯一标识 | 结论修订时保持可追踪；不得用页面位置代替标识 |
| `claim` | 当前可发布的事实或结论 | 使用完整、可验证的陈述，避免只写关键词 |
| `scope` | 适用范围与边界 | 写明产品、版本、地域、客户类型、时间或前提条件 |
| `source_url` | 主要证据来源 | 优先官方文档、标准、论文或产品公告；链接应直达原始材料 |
| `evidence_grade` | 证据等级 | A：一手权威来源；B：可信二手材料；C：待复核线索 |
| `verified_at` | 最近核验日期 | 使用 `YYYY-MM-DD`，表示实际查阅并确认过来源 |
| `review_by` | 最迟下次复核日期 | 按复核节奏计算；高风险或临时公告采用更短周期 |
| `owner` | 维护负责人 | 指向可执行复核、解释范围并推动更新的人员或角色 |
| `status` | 当前生命周期状态 | 使用 `active`、`watch`、`deprecated` 或 `replaced`；替代项必须关联新记录 |

替代历史不在公开页面展开。维护记录中应保存旧结论、变更原因、变更日期和接替它的 `claim_id`，确保既能回溯，也不会把过时说法继续暴露为现行答案。

### 面向读者的来源关联

- 全站以 `app/reference-content.mjs` 的 `sourceLedger` 作为来源标题、链接、证据类型和核验日期的唯一公开内容源；`referenceModules` 维护模块与来源的分组关系。
- 客户问答与证据卡只保存稳定 `sourceId`，并写明该来源具体支持的结论与外推边界；不得在多个页面组件中重复维护来源 URL。
- 完整来源台账只呈现在 `/references`，模块页以稳定锚点 `/references#source-<sourceId>` 回链，不重复维护来源列表。
- 修改原理、关键数据或工程边界时，必须在同一次变更中复核受影响的问答、证据卡和来源台账，并通过自动检查确认所有 `sourceId` 均可解析。
- `claim_id` 用于内部事实生命周期，`sourceId` 用于读者页面的来源关联；两者职责不同，不互相替代。

## B. 三档复核节奏

- **30 天**：模型目录、价格、配额、产品规格等高频变化事实。
- **90 天**：协议、平台能力、安全清单、基准等中频变化事实。
- **180 天**：原理、方法论、稳定架构模式等低频变化事实。

周期是最长复核间隔，不是等待时间。出现产品公告、标准版本变更、来源失效、安全事件或客户现场反证时，应立即触发复核并重算 `review_by`。

## C. 只核验变化项

巡检先离线筛选超期、来源失效、状态异常或被变更信号命中的事实，再只对变化项联网核验。若没有超期或命中项，输出“全库新鲜”报告并结束，不进行无差别重查。

巡检本身只产出报告，不直接修改在线 HTML、离线 HTML、PPT、PDF 或其他成品。确认变化影响后，按以下顺序处理：

1. 更新事实记录与替代历史；
2. 判断受影响的模块、相关模块、客户问答与云服务映射；
3. 完成内容复核与发布检查；
4. 发布新版本，并把旧结论转为历史状态。

## D. 模块发布门槛

模块不按图表、案例、卡片或问答数量验收。发布判断以内容质量为准：

- 基础概念与核心原理是否讲清；
- 适用边界、失败条件与风险是否明确；
- 与相关模块的关系是否完整且可导航；
- 云服务机会和方案组合是否具体；
- 关键事实与数据是否有可追溯证据；
- 客户问题是否给出能用于现场沟通的深度回答。

任一关键结论只有 C 级线索、证据已过 `review_by` 或适用范围不清时，不得把它作为确定性卖点发布。

## E. 内容优先，载体后置

先把内容深度、概念关系、证据链和客户回答做完整，不为控制交付包大小删减知识，也不为了填满版式增加无价值内容。内容稳定后，再从同一内容源生成在线 HTML、离线 HTML、PPT 或 PDF。

载体变化不得改变事实含义、术语中英对照、适用边界或来源关系；如果某种载体容纳不下完整解释，应提供摘要与回链，而不是静默删掉关键前提。

## 内部发布记录

每次发布至少记录版本号、发布日期、变更的 `claim_id`、受影响模块、复核人和回滚位置。公开页面可以显示版本与来源核验日期，但不得展示本规范、内部字段 schema、巡检流程或人员分工。

### Git 推送后的公开发布

Codex 对本项目执行任何 Git 推送后，必须在同一次任务中完成公开站点更新：先确认远端分支包含精确提交，再以该提交保存 Sites 版本、部署公开版本，并轮询至成功。GitHub 已更新但公开站点未更新时，发布不算完成。
