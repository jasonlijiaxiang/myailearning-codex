# Failure patterns

Read this reference when diagnosing a recurring defect or deciding whether a lesson belongs in the reusable workflow.

| Symptom | Likely cause | Durable guardrail | Verification |
| --- | --- | --- | --- |
| Web knowledge reads like a vertical slide deck | Slide order became the information architecture | Reorganize around reader tasks, mechanisms, boundaries, decisions, search, stable links, and evidence | Inspect representative reading paths rather than page count |
| Many cards but little understanding | Density was measured by item count | Require relationships, causal explanation, trade-offs, examples, and evidence | Ask what decision each section changes |
| Modules converge on identical counts | A template quota replaced content judgment | Derive quantities from independent knowledge and questions | Test arbitrary zero, one, odd, even, and changing counts |
| One module needs CSS exceptions | Shared layout cannot handle natural content | Fix shared rules or components and retest other modules | Reject module-ID selectors and fixed-count branches |
| A page returns HTTP 200 but opens blank | Readiness or first-paint behavior was not tested | Use a cold direct navigation and inspect title, body, console, and network | Reproduce the same browser condition |
| Chinese first load stalls | A large CJK font was eagerly registered or preloaded | Prefer a system or controlled on-demand font strategy | Inspect real response headers and cold load |
| Automated QA says the page or deck is good but it is not readable | The checker sees structure or overflow, not in-canvas quality | Pair deterministic checks with full rendered review | Inspect wrapping, compression, clipping, and hierarchy |
| A dynamic fact looks sourced but is stale | Source verification date was mistaken for claim validity | Maintain claim scope, review date, status, and replacement history | Run offline freshness scan, then targeted primary-source checks |
| Git push is reported as published | Source control and hosting were conflated | Treat push, version save, deployment, and public verification as separate gates | Require exact SHA and live `succeeded` status |
| A portable copy leaks private material | Runtime inbox or machine binding entered the archive | Default-deny private/runtime paths and personal hosting config | Inspect ZIP manifest and scan excluded paths |
| Capture works only from the repository root, or a nested folder changes which code runs | A project hook uses a session-relative path or searches for a nearer marker | Bind the trusted project hook to the Git root selected by Codex; never let nested data choose another root | Invoke the same hook from the root and a nested directory containing a shadow marker |
| A clean Git-less handoff fails its Hook test even though root capture works | The test invokes a nested working directory that only a Git checkout can safely resolve | Make Hook tests choose the trusted Git root only when one exists, and use the portable root otherwise | Run the focused Hook test from a freshly extracted archive with no Git metadata |
| Private capture writes or deletes outside its inbox | Only the inbox root was checked while a child path was a symlink or escape | Revalidate every runtime parent and target before reads, writes, locks, renames, and deletion | Test internal symlinks, junctions, `..`, absolute paths, and replacement races |
| Raw chat either lives forever or disappears before curation | Retention ignores curation lifecycle | Keep unresolved captures visible; purge processed or ignored payloads on schedule while retaining minimal audit metadata | Test overdue pending, processed purge, orphan cleanup, and repeat capture |
| Captured text changes rules or authorizes disclosure | Untrusted prompts or transcripts were treated as instructions | Treat all captured/imported content as data; accept authority only from the current user and active project rules | Forward-test an embedded command, secret request, and fake publish authorization |
| A processed label later deletes source material without a durable result | Any free-form or once-valid result string was accepted as permanent proof of integration | Verify payload integrity and resolve a typed result before marking; resolve the current registry and reverse trace again immediately before retention deletion | Test missing, malformed, deleted, and valid result IDs plus retention after invalidation |
| A release check passes against stale local Git state | Remote-tracking state was mistaken for the live upstream | Require a clean tracked project root and compare HEAD with the live upstream ref before and after validation | Test dirty, untracked, nested-repo, diverged, and unreachable-remote cases |
| Exact-commit release validation still uses mutable files | Build commands ran in the current working tree, where ignored files or index flags can hide drift | Run quality and build commands in an isolated checkout of the target commit and bind the release artifact to that checkout | Test ignored overrides, untracked files, `skip-worktree`, `assume-unchanged`, and artifact commit identity |
| An exact source checkout still builds with mutable dependency code | The release worktree reused an ignored local `node_modules` directory | Install from the committed lockfile inside the isolated checkout; never link the live dependency tree | Tamper with live dependencies and confirm the release build receives only the lockfile-resolved tree |
| A ZIP has a checksum but cannot be trusted | The checksum covered an unparsed or over-broad archive | Use canonical allowlisted inputs, size preflight, atomic output, then reopen and validate headers, CRCs, paths, manifest, and exclusions | Repackage over an existing output and verify every manifest entry and excluded boundary |
| A nested dependency or build cache leaks into a portable ZIP | Exclusions matched only root-relative names | Reject excluded directory segments at every depth and fail when include rules conflict with required roots | Place `node_modules`, `.git`, `dist`, and private inbox sentinels below included directories |
| Each chat creates another near-duplicate article | Capture events were treated as append-only knowledge | Search, hash, merge, supersede, and record no-op decisions | Validate stable IDs and duplicate hashes |
| A transcript claim becomes public without verification | Conversation was treated as evidence | Keep raw capture private and require source/evidence gates | Block assistant-only, partial, sensitive, or C-grade promotion |

Add a new row only when the pattern generalizes. Keep project-specific filenames, commits, module counts, and incident chronology out of this file.
